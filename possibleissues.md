# 技术风险与解决方案预案 (V1.0)

1. 网易云音乐 API 的频率限制 (Rate Limiting)

----------------------------------

**风险描述**：由于使用非官方逆向 API，短时间内发起大量请求（例如抓取 10 个歌单共 10,000 首歌的详细信息）极易触发网易云的 WAF 拦截（403 Forbidden 或验证码）。

* **方案 1：请求节流与随机延迟 (Throttling & Jitter)**
  后端 Worker 在调用 `Provider` 时，强制执行队列限速（如每秒最多 5 个请求），并在每次请求间加入 100-500ms 的随机波动，模拟真人行为。

* **方案 2：分片抓取 (Chunking)**
  不一次性获取 1000 首歌的详情，而是将歌单分片（如每片 50 首歌），处理完一片更新一次进度，让 API 请求在时间轴上均匀分布。

* **方案 3：代理 IP 池 (Proxy Pool)**
  _（仅在规模扩大时使用）_ 若单机 IP 被封禁，通过代理池随机切换请求出口 IP，这是对抗风控的最终手段。
2. “绝对时长平等”与“离散歌曲长度”的数学矛盾

-------------------------

**风险描述**：歌曲时长是离散的（如 3:20, 4:15），无法像切蛋糕一样精确切割。如果目标是每部分 60 分钟，可能凑出 59:58 或 60:15，算法需要处理这种“凑不齐”的情况。

* **方案 1：贪心洗牌算法 (Shuffled Greedy)**
  将歌曲随机洗牌后顺序累加，当加入下一首歌会超过目标时长 $t_{target}$ 时停止。这是计算开销最小、随机性最好的方案，允许存在一首歌时长的误差。

* **方案 2：余数平衡法 (Residual Balancing)**
  如果 A 歌单少了 10 秒，B 歌单多了 10 秒，在总时长层面保持平衡。通过记录每个歌单的“时长欠条”，在下一轮抽取时进行动态补偿。

* **方案 3：近似背包问题的回溯优化**
  在歌曲池中进行小范围寻找（例如在最后阶段寻找一首特定时长的歌来填补空白），以求达到秒级的精确度。
3. 登录态 (Cookie) 失效与持久化

----------------------

**风险描述**：网易云的模拟登录 Cookie 具有时效性。如果任务在执行过程中 Cookie 失效，或者用户关闭页面后任务仍在后台运行，会导致任务中断。

* **方案 1：Cookie 自动续期与状态监听**
  在 Worker 执行每一步前校验 Cookie 有效性。若失效，利用 SQLite 中存储的凭证尝试调用刷新接口（如果模拟登录库支持）。

* **方案 2：前端实时唤醒登录**
  若任务因 Cookie 失败，将任务标记为 `Need_Auth`。用户轮询进度时发现该状态，立即弹出扫码框，更新 Cookie 后 Worker 自动从断点断续。
4. 歌曲去重后的“供应塌方”

---------------

**风险描述**：如果用户输入的 3 个歌单高度重合（例如都是同一位歌手的精选），去重后可能导致某个歌单剩下的可选时长远低于其他歌单，触发你设定的“缩短总时长以保持绝对平等”策略，导致最终生成的歌单极短。

* **方案 1：预检测与智能提示 (Pre-check)**
  在正式生成前，先进行一次快速的 ID 去重扫描，计算出每个源的“最大可用净时长”，如果发现某源严重不足，立即弹窗警告并建议用户更换歌单。

* **方案 2：动态去重归属权重**
  不单纯使用“先到先得”，而是将重复歌曲归属于当前“可用余量最少”的那个歌单，从而尽量维持各方的供应平衡。
5. 内存溢出风险 (Memory Management)

-----------------------------

**风险描述**：虽然 10,000 条歌曲元数据看似不多，但在高并发、多任务并行处理时，频繁的对象创建和 JSON 解析可能导致 Node.js 进程内存激增。

* **方案 1：数据瘦身 (Data Trimming)**
  从 Provider 获取原始数据后，立即丢弃歌词、专辑封面等无关信息，仅在内存中保留 `{id, duration, sign}`。

* **方案 2：任务分批持久化**
  不要将 10,000 首歌全部保存在内存中的一个变量里，可以利用 Redis 存储中间状态，Worker 每次仅处理当前相关的部分。
